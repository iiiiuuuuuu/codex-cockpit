# 管理页控制台化 UI 优化设计

## 背景

当前管理页已经支持入口 `apikey` 管理、Token 配置、API Key 配置、配置项切换、配置删除、额度刷新和固定测试请求。页面功能完整，但视觉和信息层级更接近“配置说明 + 表单 + 表格”的叠放结构：左侧步骤说明占据较大第一屏空间，右侧把访问控制和新增配置放在同一列，底部配置列表的运行态摘要较长，日常排障时需要花额外注意力扫描重点状态。

本次优化目标是把管理页整理成更适合日常使用的运维控制台。页面应该优先回答“当前是否可用、入口是否受控、正在使用哪个配置、哪些配置异常、下一步动作在哪里”，同时保留第一次配置时需要的 Token/API Key 引导。

## 目标

- 优化管理页的信息架构，让状态、配置和操作入口更容易扫描。
- 保留现有全部管理功能，不删除按钮、入口、接口调用或隐藏设置模块。
- 保留当前 Token/API Key 新增流程，只优化布局、视觉层级和表单清晰度。
- 强化配置列表的运行态可读性，减少长摘要文本造成的视觉噪音。
- 使用更克制、专业的浅色运维控制台视觉语言，避免营销页式装饰。
- 保持单静态页面实现，不引入新的前端框架或构建流程。

## 非目标

- 不修改后端 API 协议。
- 不修改 `openai.json` 配置结构。
- 不调整账号调度、额度刷新、请求转发或测试请求的业务逻辑。
- 不新增登录、权限、审计、批量编辑等新功能。
- 不移除隐藏的 Responses 设置模块和现有相关测试。
- 不把页面改成宣传型 landing page 或大面积视觉展示页。

## 功能保留清单

实现时必须保留以下能力：

- 刷新所有账号额度状态。
- 发送固定测试请求。
- 新增随机入口 `apikey`。
- 查看入口 `apikey` 列表。
- 删除入口 `apikey`。
- 新增 Token 模式配置。
- 新增 API Key 模式配置。
- 在 API Key 模式中选择 GPT、Claude 或两者支持能力。
- 查看所有配置项。
- 显示当前使用中的配置项。
- 临时切换当前配置。
- 删除配置项。
- 展示 `auth_token` 缺失或错误提示。
- 保留隐藏的 Responses 设置模块、保存按钮和模型别名测试覆盖。

功能入口可以重新摆放，但不能消失；行为可以增加更清晰的反馈，但不能改变成功路径和错误路径的语义。

## 推荐方案

采用“状态优先的运维控制台”方案，并保留轻量配置向导。

页面由四个区域组成：

1. 顶部控制栏。
2. 状态摘要区。
3. 新增配置与访问控制区。
4. 配置列表区。

### 顶部控制栏

顶部从当前孤立按钮改为一条控制栏：

- 左侧显示页面标题，例如“Airouter 管理控制台”，以及一行短说明。
- 右侧放置全局动作：刷新、测试请求、新增随机 `apikey`。

`新增随机 apikey` 可以从访问控制面板中移动到顶部控制栏，因为它是入口访问控制的主要动作之一。移动后访问控制面板仍展示当前 `apikey` 列表和删除入口。

### 状态摘要区

第一屏增加一组紧凑状态摘要，用来快速判断当前实例状态：

- 入口 `apikey`：未配置或已配置数量。
- 配置项数量：当前可用上游配置数量。
- 当前激活：当前使用中的配置编号；如果没有临时切换，则显示默认状态。
- 健康状态：从已有运行态摘要中提取可读状态，突出 `quota`、`refresh`、`timeout`、`401` 等异常线索。

这些摘要只使用已有快照数据计算，不请求新接口。

### 新增配置与访问控制区

中间区域采用左右布局：

- 左侧为新增配置项。
- 右侧为访问控制。

新增配置项继续保留 Token/API Key 模式切换。Token 模式展示 AuthSession JSON 文本框和简短提示；API Key 模式展示 Base URL、API Key、备注和支持能力选择。模式切换应有明确选中态，字段提示要短，避免重复左侧大段说明。

访问控制面板展示入口 `apikey` 状态和列表。空状态文案保持清楚，但视觉上更轻，不占据过多高度。删除按钮保留红色语义。

现有左侧三步 guide 不再作为大侧栏占据第一屏。它可以压缩为新增配置区内的简短帮助文案，或在表单旁边以轻量提示呈现。重点是让日常状态和操作优先于说明文档。

### 配置列表区

配置列表继续使用表格，因为该页面需要比较多个账号或上游。表格优化方向：

- “当前使用”用状态标签突出。
- 上游配置或 `account_id` 单元格允许换行，保持长 ID 可读。
- `description` 保持普通文本展示。
- 运行态摘要拆成更易扫读的状态片段；如果不能可靠解析，则保留原始摘要但降低视觉噪音。
- 操作列保留“切换/当前”和“删除”，按钮尺寸稳定，避免换行挤压。

表格在窄屏下仍可横向滚动，不强行牺牲信息密度。

## 视觉方向

采用克制、清晰、工具感强的浅色控制台风格：

- 背景使用浅冷灰或雾白，减少当前暖色玻璃拟态的装饰感。
- 主文字和主要按钮使用深墨蓝或近黑，保证对比度。
- 状态色只用于真实状态：成功/当前使用用绿色或青绿色，警告用琥珀色，危险操作用红色。
- 卡片圆角控制在 8-14px，按钮和输入框保持稳定尺寸。
- 适度使用边框、阴影和分区背景建立层级，不使用大面积装饰图形。
- 字体继续使用系统中文友好的 sans/mono 栈，避免引入外部字体依赖。

整体风格应像可靠的本地运维工具，而不是产品官网。

## 前端实现边界

预计主要修改：

- `public/config-admin.html`
- 必要时补充 `public/config-admin.js` 中纯展示 helper
- `test/config-admin.test.js`

实现可以增加少量纯函数，例如：

- 统计状态摘要。
- 从运行态摘要文本中提取状态标签。
- 生成配置列表展示模型。

这些 helper 应保持无 DOM 依赖，便于 Node 原生测试覆盖。

不引入 React、Vue、Tailwind、图标库、打包工具或额外运行时依赖。

## 错误处理与状态

现有错误处理语义保持不变：

- `auth_token` 错误时继续展示明确错误提示，并阻止误导性的空列表展示。
- 请求失败继续显示后端返回的可读错误。
- 添加、删除、刷新、测试请求过程中继续禁用对应按钮，避免重复操作。
- 删除入口 `apikey` 和删除配置项继续保留确认弹窗。

UI 优化可以让错误提示更醒目，但不能吞掉原始错误信息。

## 响应式设计

桌面宽度下优先展示完整控制台：

- 顶部控制栏横向排列。
- 状态摘要横向排列。
- 新增配置和访问控制左右排列。
- 配置列表横向完整展示或在容器内滚动。

窄屏下：

- 顶部动作换行排列。
- 状态摘要变成两列或单列。
- 新增配置和访问控制改为上下排列。
- 表格保持最小宽度并在容器内横向滚动。

所有按钮文字、状态标签和表单提示不得相互遮挡。

## 测试策略

自动测试：

- 保留并更新现有管理页静态 HTML 断言。
- 为新增的状态摘要或运行态展示 helper 添加纯函数测试。
- 确认隐藏的 Responses 设置模块仍在 DOM 中且保持 hidden。
- 确认激活控制、删除控制和 API Key 表单相关标记仍存在。

手工/浏览器验证：

- 使用本地管理页确认功能入口都可见。
- 截图检查桌面布局的信息层级和视觉效果。
- 在较窄视口下检查表单、顶部动作和表格滚动。
- 确认当前已有配置项能正常显示当前使用、切换和删除按钮。

## 验收标准

- 页面第一屏能清楚看到入口 `apikey` 状态、配置项数量、当前使用状态和主要操作。
- 新增 Token/API Key 配置流程比当前页面更清楚，不再要求用户手写 API Key 配置 JSON。
- 配置列表比当前页面更容易扫描，运行态摘要不会压垮整行视觉层级。
- 所有功能保留清单中的能力都仍可从页面触达。
- `npm test` 通过。
- 浏览器桌面截图无明显重叠、溢出或空白异常。
- 窄屏检查无按钮文字挤出、表单遮挡或布局崩坏。
